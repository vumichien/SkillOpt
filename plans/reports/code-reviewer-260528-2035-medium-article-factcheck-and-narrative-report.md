## Discrepancies / Unsupported Claims

### HIGH severity

- **D1. DeepSeek input price is wrong by 10×.**
  Article (lines 9, 48, 84): `$0.014 / $0.28 per 1M tokens`. DeepSeek official pricing (deepseek-chat, cache miss) is **$0.14 / $0.28**. The output number is correct; the input number is missing a digit. Re-check the cost claim: at $0.14 input × 0.200 M + $0.28 output × 0.023 M = $0.028 + $0.0064 ≈ **$0.034**. The "~$0.01" / "one cent" / "$0.01 in the headline" all become **~3 cents**.
  - Headline `I Spent $0.01…` and TL;DR `~1 cent of DeepSeek spend` are unsupported. Recommend rewriting to "~3 cents" or "under a nickel" — still a dramatic number, but defensible.
  - **Note:** DeepSeek also has cache-hit pricing ($0.014/1M for cache hits). If the author intended that, it should be labeled explicitly ("cache-hit price; cache-miss is $0.14"). Currently the article makes a flat claim that doesn't survive scrutiny.

- **D2. v3 skill sizes off by ~1%.** Article claims `154 → 374 → 1707 → 3628 B` and "154 B → 3.6 KB".
  Actual on-disk sizes: `skill_v0000.md = 157`, `v0002 = 380`, `v0003 = 1728`, `v0004 = 3666` (final accepted). The article values are the `skill_len` fields from `history.json` (which are character counts of the in-memory string, e.g. `history.json:73 skill_len:154`, `:143 skill_len:374`, `:215 skill_len:1707`, `:289 skill_len:3628`). Both are defensible, but the article mixes the units: it says "**B**" (bytes) for numbers that are actually character lengths. Either:
  - (a) Re-cite as character counts: `154 → 374 → 1707 → 3628 chars`, or
  - (b) Use real on-disk byte counts: `157 → 380 → 1728 → 3666 B` (CRLF inflates by ~1%).
  Same issue for v2: article says `157 B → 547 B`; on-disk `skill_v0000.md = 157 B`, `skill_v0006.md = 547 B`. Coincidentally consistent for v2 but inconsistent with v3 — pick one convention.

Consider "+4 pp" once with a footnote, then "%" thereafter.

- **N3. Scaffolded "What's coming next" section.** Four experiments laid out with hypothesis-up-front. Good structure, but:
  - Tables list "_coming_" placeholders → readers landing late may think the article is stale. Add a single dated line at the top of the section: *"Last updated: 2026-05-28. Tables fill in as experiments complete."* This sets expectations.
  - Experiment 3 table conflates "CSQA-trained (v3)" row showing `0.740` for CSQA test (correct) with the new ARC entries — fine. But the v4 row uses the column ordering `CSQA | ARC-Challenge` while the row label is `ARC-trained (v4)` — Experiment 3 actually trains on ARC, so the diagonal hits ARC-test. Double-check labeling is unambiguous before publishing the updates.

- **N4. Hook strength.** First two lines (the headline + the > quote-style intro) — both strong. The "12 minutes, one cent, three honest findings" closer is excellent. Keep modulo D1.
